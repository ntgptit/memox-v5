Mx Kit Screens · 12

Mx Composition Recipe Gallery

Tokens make a screen consistent; composition makes it correct. A screen can use every --mx-* token perfectly and still be wrong — two primary actions, a card nested in a card, or a due amber shown as error red. Each recipe below pairs a good Mx composition against a concrete bad one, annotates the spacing rhythm, typography hierarchy, and action hierarchy at play, and lists the common mistakes it guards against. All mocks are token-styled; where useful a data-theme="light" pane shows the same rule holds in both themes.

How to read this The DO pane is a real Mx composition built from MxCard, MxActionButton, MxListRow, etc. The DON'T pane is the same content composed wrong. The mistake is in the arrangement, not the tokens — both panes draw from the same palette.

Spacing rhythm good: gutter 16 · item gap 12 · row internal 12. bad: random 8 / 4 / 20 padding per row, no shared rhythm.

Flat option rows, one radius level, a grab handle to signal the sheet, and no destructive action buried among navigation choices. A "Delete deck" would live in its own separated destructive variant — never mixed into the play list.

Spacing rhythm good: sheet padding 20 · option gap 12 · handle→title 12. bad: nested paddings stack (16 outer + 12 inner) with no consistent gap.
Typography good: title-medium heading, body option rows. Action hierarchy good: cancel is ghost, no competing primary; bad: a destructive row sits inline with benign options.
Nesting good: flat rows on the sheet surface. bad: MxCard inside MxCard inside the sheet — three stacked surfaces, muddy depth.
  • Card-in-card nesting on the sheet.
  • Destructive option inline with normal options.
  • Missing grab handle / title.
  • Rows too small to tap (below 44 target).

7 · Empty / error

MxEmptyErrorScreenRecipe — centered icon → title → message → recovery action. No-due is positive/neutral, not an error. The message explains what happened; the recovery action is clear; finalize-failed preserves data and offers retry.

Spacing rhythm good: generous 48 top/bottom breathing room for a centered state. bad: cramped 16 — the empty state feels like a squished error toast.
Typography good: title-medium + body message, calm. bad: same size but red.
Colour / semantics good: no-due uses srs-noDue/feedback-correct green and a check — a good outcome. bad: painted with feedback-wrong red and an ✕ — a normal, positive state framed as a failure.
State good: a recovery action ("Browse decks"). bad: no recovery action at all — dead end.
  • No-due (a good state) styled as an error.
  • Message that doesn't explain what happened.
  • Missing recovery / retry action.
  • finalize-failed that loses data instead of preserving + retry.

Composition is enforced in review

React Native Composition rules are enforced in review, not by tokens alone — a screen can be token-correct and still composition-wrong. Build every screen from a recipe: use MxActionButton action hierarchy (≤1 primary per region; destructive never styled as a normal primary), MxCard with no card-in-card nesting, one spacing rhythm from MxSpacing, and design every applicable state (loading / empty / error / no-due / finalize-failed). Semantic colour follows intent: srs-due amber ≠ feedback-wrong red, state-selected violet ≠ feedback-correct green.
