Mx Kit Screens · 10

Mx State Gallery

Every interactive and learning surface in MemoX has a defined visual for each state it can enter. A state is never a colour a screen invents on the spot — it is a state/variant prop the component resolves to var(--mx-color-state-*), --mx-color-feedback-*, --mx-color-srs-*, or --mx-color-destructive-* tokens. This page is the master vocabulary: what each state means, where it is allowed, where it is forbidden, and the dangerous confusions to avoid — shown in dark and light together.

Source of truth The state token values shown resolve from tokens/colors.css (--mx-color-state-* · --mx-color-feedback-* · --mx-color-srs-* · --mx-color-destructive-*). Every state below already has both a dark :root value and a light [data-theme="light"] value — a component never picks a new colour per state.

Canonical state galleries — dark & light

The base kit ships a single specimen of every interaction and learning state, rendered once per theme. These are the source specimens; everything below is the vocabulary and the per-component breakdown that cites them.

State vocabulary — meaning, allowed, forbidden, token

Read this table as a contract. The Forbidden usage column is the important one: most state bugs are a correct token used to mean the wrong thing. The confusions that break MemoX are called out explicitly — selected ≠ correct, wrong ≠ destructive, warning (amber / due) ≠ error (red), no-due ≠ error, and unavailable ≠ disabled ≠ broken.

default Resting, interactive, nothing chosen yet. Every idle button, card, row, option. Do not tint it toward selected/correct to "hint" the answer. --mx-color-surface-card
pressed Momentary touch-down feedback. Active press on any tappable — darker fill / primaryPressed. Never persist after finger-up; pressed is not selected. --mx-color-action-primaryPressed

disabled Exists but not actionable right now (precondition unmet). Submit before an answer is picked; greyed control. disabled ≠ unavailable ≠ broken. Not for missing features or errors. --mx-color-state-disabledSurface
loading Work in progress; outcome pending. Button spinner, skeleton surface while fetching. Do not freeze into disabled; do not hide that it is temporary. --mx-color-state-loadingSurface
empty Valid state with no data yet — an invitation, not a fault. MxEmptyState: no decks, no due cards, no results. empty ≠ error. Never render empty in red or as a failure. --mx-color-text-tertiary
error Something failed and needs recovery. MxErrorState, failed load, invalid input — red feedback. Not for "due", "wrong answer", or an empty list. --mx-color-feedback-wrong
warning Attention / caution — amber, not a failure. Due badge, "cards waiting", non-blocking caution callout. warning (amber) ≠ error (red). Due is not danger. --mx-color-feedback-warning
unavailable Feature/entry not offered in this context (by design). Locked setting, mode not applicable to this deck. unavailable ≠ disabled ≠ broken. Not a transient block, not a crash. --mx-color-state-unavailableSurface
Learning & grading states
StateSemantic meaningAllowed usageForbidden usageKey token
correct Answer graded right — green feedback. Post-grade answer option, match tile, study card result. Never show before grading; correct is not "selected". --mx-color-feedback-correct
wrong Answer graded incorrect — red feedback. Post-grade wrong option; the missed correct answer is still shown as correct. wrong ≠ destructive. A wrong answer deletes nothing. --mx-color-feedback-wrong
completed Item resolved and removed from play (matched / finished). Matched pair fades to done; finished session item. Not the same as correct; a completed pair is not "graded green" forever. --mx-color-feedback-correct
revealed Answer shown to the learner (self-graded flow). Study card flipped to its meaning before self-rating. Reveal is neutral — do not colour it correct/wrong pre-rating. --mx-color-srs-active
forgotten Recalled poorly → SRS lapse. "Again" self-rating; lapse pill in review history. Not destructive red; lapse is srs-lapse coral, not delete. --mx-color-srs-lapse
remembered Recalled well → interval advances. "Good"/"Easy" self-rating; progress toward mastered. Do not reuse the answer-correct token as an SRS lifecycle colour. --mx-color-srs-active
no-due Nothing due — a calm, good resting state. "All caught up" badge / empty due-queue → srs-noDue green. no-due ≠ error. Zero due is success, never a red empty fault. --mx-color-srs-noDue
destructive Irreversible data loss — delete deck / reset progress. Destructive MxActionButton + MxConfirmDialog. Never style a normal primary as destructive; never for "wrong answer". --mx-color-destructive-text
high-risk Destructive and hard to undo — needs extra friction. Reset all data: destructive styling + typed confirm / hold-to-confirm. Do not gate everyday deletes behind high-risk friction (fatigue). --mx-color-destructive-border
Name the intent, then let the component resolve the token: selected for a choice, correct/wrong only after grading, warning/srs-due amber for "needs review", destructive only for data loss.
Blur the dangerous pairs: green a mere selection (selected→correct), red a wrong answer as if it deleted data (wrong→destructive), red a due badge or an empty list (warning/no-due→error), or grey an unavailable feature as if it were broken.

The confusions that break MemoX

These four pairs share a hue or a feeling but carry opposite meaning. Getting them wrong teaches the learner the wrong lesson or scares them off a safe action.

selected ≠ correct. A chosen answer is violet (state-selected). It only turns green (feedback-correct) after grading — never pre-colour a selection as if it were right.
Paint a selected option green so the learner can guess the answer from the colour before submitting.
wrong ≠ destructive. A wrong answer is red feedback-wrong and destroys nothing. Destructive red (destructive-*) is reserved for delete / reset.
Reuse the destructive button style for a wrong answer, implying the mistake cost the learner their data.
warning (amber / due) ≠ error (red). Due cards use amber srs-due/feedback-warning — a nudge, not a failure. Errors are red feedback-wrong.
Render a due badge or "cards waiting" in red so a normal review queue looks like something crashed.
no-due ≠ error and unavailable ≠ disabled ≠ broken. "All caught up" is green srs-noDue success; unavailable is a calm not-offered surface, disabled is a transient block.
Show an empty due-queue as a red error, or grey an unavailable Pro feature so it reads as a bug.
React Native Every Mx component takes a state / variant prop that resolves to tokens inside the component — e.g. <MxAnswerOption state="correct" />, <MxActionButton variant="destructive" />. Screens never branch raw colours per state (state === 'wrong' ? '#ff8a8a' : …) and never invent a new state colour. If a state has no token, add one to tokens/colors.css — do not hardcode.
