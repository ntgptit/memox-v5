Mx Theme Gallery

MemoX ships two first-class themes: MxDarkDefault (the primary direction — a calm, purple-tinted deep aubergine surface that keeps attention on the card) and MxLightDefault (a clean, white-first, production-ready surface — not a mechanical inversion of dark). Every semantic token defines a dark value on :root and a light value under [data-theme="light"]; a screen consumes the role and the theme resolves the value. Below, each demo is shown in both themes side by side using the same markup — the theme comes only from the data-theme attribute, never from a duplicated Dark*/Light* component.

Source of truth Dark values resolve from :root; light values from [data-theme="light"] in tokens/colors.css · elevation.css. The two panes in every .mx-duo render identical HTML — only the theme scope differs.

Theme behaviour

Both themes are first-class. Dark is the primary direction and light is complete and production-ready — a designed surface, not a mechanical negative of dark. The rules below are what keeps the two themes coherent.

Define every semantic token in both themes — a dark value on :root and a light value under [data-theme="light"] — and consume the role so a screen renders correctly in either.
Invert colours mechanically (flip dark to light by negating hex) or leave a role defined in only one theme — light is designed white-first, not a computed inversion.
Use one MxCard that reads theme tokens; toggle the theme by setting data-theme="light" on an ancestor.
Create MxDarkCard/MxLightCard (or any theme-specific duplicate component) — same markup, different theme tokens only.
Theme switch is visual only Switching MxDarkDefault ⇄ MxLightDefault is a display preference. It never mutates learning progress, study-session state, or SRS box/due data — those live in cards, card_reviews, and the session tables, untouched by the appearance setting.

React Native theme handoff

In the app, both themes resolve through a single provider. MxThemeProvider wraps the app and holds an MxThemeMode; MxThemeResolver maps that mode to the concrete dark/light token set; useMxTheme() reads the resolved theme inside a screen. Tokens are resolved before a component renders, so a screen never branches on the OS colour scheme itself.

Resolve MxThemeMode.system in MxThemeResolver at the root, then read tokens with useMxTheme().
Call raw useColorScheme() inside a product screen and branch on 'dark'/'light' — that duplicates the resolver and drifts from the tokens.
React Native The appearance setting sets MxThemeMode on MxThemeProvider; MxThemeResolver resolves it to the dark or light token set once, and every Mx component reads its colours from useMxTheme(). Screens never hold a hex literal and never run a per-screen useColorScheme() branch.
