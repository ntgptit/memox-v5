Mx Design Kit Screens

The internal design-kit screens that document, demonstrate, and validate the Mx Base Mobile Design Kit — the visual source of truth for every future MemoX product screen and its React Native implementation. Every page here composes the real mx tokens and Mx components already in this kit; nothing is a second, parallel kit.

Use these pages to design new screens from existing tokens, components, layout templates, and composition recipes. Map every design decision to an Mx component and mx token before drawing anything custom.
Treat these as final product screens, invent new business behavior, hardcode a colour/spacing/font, or create a screen-specific reusable component. Sample frames exist only to demonstrate the kit.

What the kit is — and is not

The kit is

  • The single source of truth for tokens, themes, type, spacing, components, states.
  • A foundation — vocabulary (tokens + components) plus grammar (composition rules).
  • Dark-first, light production-ready — both documented equally.
  • A React Native handoff contract for implementation agents.

The kit is not

  • Final Deck / Study / Settings / Statistics product screens.
  • A place to invent business rules (Box 0 activation, 5-mode flow, read-only Review Words).
  • A second design kit — it references the base kit, never duplicates it.
  • A React Native implementation — it is the design + handoff, not the app.

How to use these screens

If you design future product screens

Start at the layout template that matches your screen (Page 11), copy its region structure, then fill regions with Mx components from the component galleries (05–09). Check spacing/typography/action hierarchy against the composition recipes (Page 12) and the mini golden samples (Page 13). Design every applicable state from the State Gallery (Page 10). Validate against the QA checklist (Page 15) before handing off.

If you implement in React Native

Read the React Native Handoff Gallery (Page 14) first: it maps every kit name to a shared RN component, RN primitive, and token module, and lists the guard rules. Resolve tokens through one MxThemeProvider / MxThemeResolver; never branch on useColorScheme into raw colours inside a screen.

Two rules every page enforces

Dark / light parity

Every semantic token defines a dark and a light value; every component works in both. There is exactly one MxCard — never MxDarkCard/MxLightCard. Switching theme is a visual preference only; it never mutates learning data. These pages show both themes side by side (the duo pattern) so parity is provable, not assumed.

No raw styles

No raw colours, no raw spacing, no arbitrary typography. Every visual value resolves to an mx token. If a value you need isn't a token, the fix is to add a token to the kit — not to hardcode it in a screen. This page's own chrome obeys the rule.

Governance summary

This kit is the living source of truth. New screen style requires updating the kit first. New reusable components enter the kit before screen use. Screen-specific components stay unprefixed until promoted. Full rules in guidelines/governance.md and the change-flow in composition-governance.md.

React Native target note The real repo is Expo SDK 57 · React Native · TypeScript · NativeWind. The React Native handoff on Page 14 is the single authoritative handoff for implementation.
